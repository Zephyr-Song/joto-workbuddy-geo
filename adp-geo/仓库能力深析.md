## 1. 一句话定性

- `package.json` 名为 **`joto-gtm-workbench`**，内部文档自称 **「JOTO GTM 内容工作台」** —— 它就是任务文档里反复说的"工作台"，是整个 ADP GEO 战役的**执行载体**。
- 本质：面向企业内容增长团队的 **GEO（生成式引擎优化）自动化内容生产闭环**，不是 AI 写作玩具。核心链路：
  `产品/知识治理 → 月度内容矩阵 → Evidence Gate → 批量生成+质检 → 发布排程+人工确认 → URL/渠道回传 → GEO 监控/主动测试 → 月度复盘`
- 当前状态：**V4 可运行闭环 + V5 月度架构切换中**。README 对"已实现 / 待验收 / 不能承诺"区分非常诚实（明确写了不伪造发布、不伪造 GEO 命中）。
- 归属：仓库 owner 是 `1736280642-Star`（某位团队成员个人账号，非 JOTO 官方 org）。协调人让大家"自行部署"= 每人 clone 这份共享基座本地跑。

## 2. 技术栈与运行方式（关键点）

| 项 | 内容 |
|---|---|
| 框架 | Next.js 14（App Router / Route Handlers）+ React 18 + TypeScript |
| UI | Ant Design 5 |
| 数据 | MySQL 8（`mysql2`）+ **本地 JSON 状态兜底** |
| 知识库/RAG | 抓取（XCrawl / 代理）+ Chunk + Embedding + OpenSearch 检索 |
| AI Provider | **qwen / deepseek / doubao（豆包）**，走 OpenAI 兼容 `/chat/completions` |
| 发布 | 微信公众号草稿 bridge（可用）；CSDN/掘金/知乎 adapter 合同已就绪 |

**最关键的一点：可以无 MySQL 本地试运行。**
- 默认本地状态文件 `data/workbench-state.json`，单机 JSON 模式即可跑通 V4 闭环（知识库、GEO 测试、博客监控、发布排程等）。
- 只有 V5 月度底座 / 知识治理持久化才需要 MySQL。`npm run dev` 即可在 `http://127.0.0.1:3047` 起来。

**AI Provider 真实接线情况（看 `src/lib/ai-provider.ts`）：**
- 支持：`qwen`(DASHSCOPE) / `deepseek`(api.deepseek.com) / `doubao`(ark.volces.com)
- 平台→模型映射：`DeepSeek → deepseek`，`豆包 → doubao`，**其他一律默认 `qwen`**。
- ⚠️ 坑：README 里提到 ChatGPT / OpenAI，但代码里**没有 OpenAI 接线**。GEO 测试若选 "ChatGPT" 平台会落到 `qwen`。任务文档 A3 要求"豆包 + DeepSeek"，这两者都真实可用。

## 4. 真实可用 vs 待验收（按 README 状态表）

**✅ 已实现、可本地试运行：**
- V4 内容生产、质检、URL 回填、GEO、复盘页面
- V5 月度数据底座（MySQL 表：MonthlyPlan/策略包/矩阵/生产池/产物引用）
- V5 知识治理后端首批垂直切片（产品实体、资料、事实、冲突、证据缺口、规则包、G0-G6 Gate、审计、版本、幂等）
- RAG / EvidencePack / Final Evidence Gate 单篇代码链路
- 单篇正式生成（1 个矩阵项真实 Retrieval→EvidencePack→正文→DraftVersion）
- 发布排程 P0 状态层、适配器合同、失败分类、人工接管
- GEO 监控与主动测试（V4 可运行）

**⚠️ 未完成 / 待真实验收：**
- **四平台真实正式发布**：微信草稿 bridge 可用；CSDN/掘金/知乎浏览器自动化仍需真实验收
- **整月批量生成 + 完整 Eval Runner**：不在本轮范围（仅单篇）
- **OpenSearch / Embedding 真实检索**：代码已接，需配置后验收
- Workflow Agent：规则/调度边界已设计，但**不能自主批准策略、规则包、风险例外或正式发布**

**🚫 明确不能做的（红线，仓库自身强制）：**
- 不把本地 fallback 草稿伪装成真实 Provider 生成
- 不把"平台草稿已创建"当作"正式发布成功"
- 不伪造 GEO 命中（无 AI 配置时只返回 `pending_config`）
- 不绕过 Evidence Gate / 硬规则 / 人工审批

## 5. 对你们团队"自行研究和部署"的意义

协调人说"大家自行研究和部署仓库" = 每人 clone 这份 repo 本地拉起熟悉。结合当前处于**阶段0（正式5问未发）**，现在能做的：

1. **本地拉起**（无需 MySQL）：
   `git clone` → `npm install` → `npm run dev`（默认 `127.0.0.1:3000`，或 `--port 3047`）
2. **A1 产物直接入库**：把你写的「官方资料合集 / 竞品占位 / 单题内容规范清单」通过知识库导入（URL 或 Markdown/PDF/DOCX）变成 SourceAsset；把"内容边界/禁忌要点"转成**产品表达规则包草稿**（需人工确认才生效）。
3. **预建规则包**：这正是 A1 的"内容边界+禁忌"价值落点——比纯文档更可被 A2 复用。
4. **跑 GEO baseline（exploration 模式）**：配一个 deepseek + 一个 doubao 的 key，即可对"WorkBuddy 服务商怎么选"之类真实提问做主动测试，直接印证"JOTO 0% 可见度 / 竞品占位"的基线——这正好是 A3 在正式题发布前的探索式预研。
5. **不急着接 MySQL / 真实发布**：阶段0 用本地 JSON 足够；等正式5问发布、且 JOTO 官方专项页建好后再上 MySQL + 真实发布。

## 6. 部署速览（无 MySQL 先跑通）

```powershell
git clone https://github.com/1736280642-Star/gtm-content-growth-workbench.git
cd gtm-content-growth-workbench
npm install
# 可选：建 .env.local 只填 AI key（deepseek/doubao）即可跑 GEO 测试
#   DEEPSEEK_API_KEY=... DEEPSEEK_MODEL=...
#   DOUBAO_API_KEY=...   DOUBAO_MODEL=...
npm run dev -- --hostname 127.0.0.1 --port 3047
# 打开 http://127.0.0.1:3047
```

想用 V5 月度 MySQL 底座再跑：
```powershell
npm run check:mysql      # 先配 MYSQL_* 环境变量
npm run init:mysql
node scripts/init-v5-monthly-schema.mjs --plan
node scripts/init-v5-monthly-schema.mjs
```

## 7. 风险与坑（给团队提个醒）

1. **ChatGPT 没接线**：GEO 测试选 ChatGPT 会落到 qwen；任务 A3 要求"豆包+DeepSeek"，这两个真可用，别选错平台。
2. **本地 JSON 是单人模式**：不支持多人并发/生产级一致性。多人协作要上 MySQL + 独立 Worker。
3. **微信草稿 ≠ 正式发布**：bridge 只创建草稿箱，需人工在公众号后台预览发布再回填 URL。CSDN/掘金/知乎更只是 adapter 合同，未真实验收。
4. **JOTO 零可见度的根因不在工具**：是缺"可抓取官方专项页"——工作台再强，没有源也引不到 JOTO。建页仍是最高优先。
5. **仓库是个人账号下的共享基座**：非官方 org，后续若团队要长期维护，建议统一 fork/迁到团队归档仓库。
6. **密钥纪律**：`WECHAT_MP_*`/`*_COOKIE`/`*_API_KEY` 只进本机 `.env.local`，绝不进 Git/聊天/截图（README 反复强调）。
